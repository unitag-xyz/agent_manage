# agent_manage Product Notes

## 1. Overall Goal

本项目服务于 Unipay 的 OpenClaw 管理体系。

目标不是只做一个脚本，而是支撑以下完整链路：

- 模板作者提供自己的 OpenClaw 母版能力
- Unipay 负责销售、成交、交付和管理入口
- 买家支付后自动获得一个可直接使用的 `claw instance`
- 买家进入 `My Claw` 页面进行管理
- `agent_manage` 负责服务器内的实例执行和配置管理

当前阶段先做 MVP，重点是把整体模型和职责边界梳理清楚，不急于一次性开放全部自动化和模板市场能力。

---

## 2. Core Product Objects

### 2.1 Claw Instance

`claw instance` 是买家最终拿到并实际使用的运行实例。

当前理解：

- 一个订单当前对应一个 `claw instance`
- 当前可以理解为一个新的 OpenClaw 主机实例或其实例空间
- 未来再扩展成“一个 claw 下有多个 agent”的模式

实例页当前 MVP 关心：

- 实例名称
- 实例有效期
- 实例描述
- 续费按钮
- 恢复按钮（用于用户用崩后恢复到上一次配置）
- 微信链接
- TG 链接
- 控制台链接

补充状态建议：

- 开通中
- 可用
- 绑定中
- 失败
- 已过期
- 恢复中

### 2.2 Template Project

母版不能直接等同于运行中的实例目录。

更合理的定义是：

- 母版是一个结构化的“模板工程”
- 母版可以包含：
  - soul / 人设 / 做事规则
  - skills / 能力 / 程序
  - 默认配置
  - 资源文件
- 母版应当是可审核、可版本化、可发布的对象
- 运行中的实例只可作为“导入草稿来源”，不应直接作为可售母版

### 2.3 Template Release

真正绑定商品和交付给买家的，应当是 `template release`，而不是一个运行时实例快照。

`template release` 的特点：

- 有明确版本
- 经过平台审核
- 可绑定到 Unipay 商品
- 买家买到的是基于此版本创建的实例
- 作者后续修改不直接影响已售实例

## 3. High-Level System Structure

整体分为 4 层：

### 3.1 网站 / Unipay 层

安装位置：

- Unipay 前端

职责：

- 商品销售
- 支付和订单
- `My Claw` 页面
- 用户进入实例管理页
- 展示和维护微信 / TG / 控制台链接
- 后续支持 soul / agent / skill / 用量等管理入口

MVP 中：

- 商品页前端流程不改
- 支付成功后进入 `My Claw`
- 订单 cell 增加“去 Claw”按钮

### 3.2 服务器管理层

安装位置：

- Unipay 后台

职责：

- 管理 AWS 服务器
- 管理镜像和实例初始化
- 自动建机和部署基础环境
- 注册服务器元信息

MVP 方向：

- 支付成功后先建 `claw instance` 记录
- 状态为“开通中”
- 异步创建 AWS 主机并部署 OpenClaw 环境

### 3.3 OpenClaw Server API 层

安装位置：

- claw 实例主机

职责：

- 对外接受服务器控制命令
- 调用本机 `agent_manage`
- 统一鉴权、日志、审计和标准返回

### 3.4 agent_manage 层

安装位置：

- claw 实例主机

职责：

- 运行在 OpenClaw 服务器内部
- 作为本机执行器
- 负责创建实例、渠道绑定、恢复配置、升级分析等
- 不负责商品发布规则
- 不负责模板市场权限判断
- 不直接负责 AWS 管理

## 4. Current MVP User Flow

### 4.1 下单和交付

1. 用户在 Unipay 商品页下单
2. 商品页前端不改
3. 支付成功后系统立即创建一个 `claw instance` 记录
4. 该记录初始状态为“开通中”
5. 系统异步执行：
   - 创建或分配 AWS 主机
   - 部署 OpenClaw 运行环境
   - 注册 OpenClaw Server API
   - 调用 `agent_manage` 初始化实例
6. 成功后实例状态变为“可用”
7. 用户进入 `My Claw` 页面

### 4.2 My Claw 页面

当前先做单实例详情页，不急于设计 list 页面。

MVP 页面字段：

- 实例名称
- 实例有效期
- 实例描述
- 续费按钮
- 恢复按钮（用于用户用崩后恢复到上一次配置）
- 微信链接
- TG 链接
- 控制台链接

补充建议：

- 实例状态
- 创建时间
- 到期时间
- 最近操作时间
- 开通状态说明

### 4.3 微信链接流程

更正后的结论：

- 前端自己负责出二维码或完成登录流程
- 后端 / `agent_manage` 不负责二维码生成
- `agent_manage` 只负责绑定 key 和写入配置

流程：

1. 用户在前端完成微信侧流程
2. 前端把得到的 key 提交给后端
3. 后端调用 `agent_manage` 绑定该 key
4. `agent_manage` 写入微信配置
5. 测试绑定后是否需要 reload / restart
6. 页面显示绑定成功 / 失败状态

### 4.4 TG 链接流程

当前方向：

- 用户提供 Bot Token 和必要账号信息
- 后端调用 `agent_manage` 建立 TG 绑定
- 测试写入后是否需要 reload / restart

## 5. agent_manage Responsibilities

`agent_manage` 当前和未来重点职责如下。

### 5.1 create_instance

创建实例不是简单创建空 agent，而是：

- 新服务器创建完成后
- 按商品绑定的母版创建一个新实例
- 母版中包含 soul、skills、默认配置等
- 再按订单或用户信息覆盖必要字段

当前重点：

- 支持从商品绑定母版创建新实例
- 初始化实例目录和配置
- 注册到 OpenClaw

### 5.2 bind_wechat

职责：

- 接收微信 key
- 绑定到指定实例
- 写入配置
- 测试是否需要 reload / restart

### 5.3 unbind_wechat

职责：

- 删除指定实例的微信绑定
- 清理相关配置
- 测试是否需要 reload / restart

### 5.4 bind_tg

职责：

- 接收 Bot Token 和必要 TG 参数
- 写入实例配置
- 建立 TG 与实例的绑定关系
- 测试是否需要 reload / restart

### 5.5 unbind_tg

职责：

- 删除指定实例的 TG 绑定
- 清理配置
- 测试是否需要 reload / restart

### 5.6 restore_backup

当前恢复方向偏向：

- 仅恢复实例配置
- 不恢复 workspace 内容
- 用于恢复 soul、skills、渠道绑定等配置

### 5.7 template / release support

后续需要支持模板升级、模板发布辅助、实例版本记录等能力。

## 6. Template Architecture

### 6.1 不建议直接从运行实例目录做可售母版

原因：

- 容易混入运行态脏数据
- 容易携带登录态、缓存、密钥
- 容易引入侵权和恶意程序风险
- 无法清晰管理版本和审核

### 6.2 建议的模板结构

母版应理解为：

- 配置层 + 能力层 的组合发布物

配置层可能包括：

- soul
- 人设
- 做事规则
- 文案
- 默认配置
- 资源

能力层可能包括：

- skills
- 程序
- 工具脚本
- 对外接入能力

### 6.3 模板与技能分层

建议把模板和 skills 分开管理：

- 模板声明自己依赖哪些 skills
- skills 作为独立可审核、可版本化的能力资产
- 模板发布时绑定 skill 版本，而不是直接混一个大目录

这样更利于：

- 审核
- 下架
- 回滚
- 定位问题
- 权限控制

### 6.4 从实例提取的正确定位

“从现有实例提取母版”只应作为：

- 内部导入工具
- 草稿生成工具

不应作为：

- 自动公开商品生成机制
- 普通用户直接发布商品的入口

正确链路应是：

- 实例 -> 草稿母版工程 -> 审核/清洗 -> template release -> 商品

## 7. Product Publishing Rules

当前判断：

- Unipay 本质是帮助别人赚钱的平台
- 平台未来允许第三方模板作者通过 Unipay 销售带有自己母版的 OpenClaw 实例
- 但 MVP 不开放作者自动上传母版并直接创建商品
- MVP 先走后台人工代发布

### 7.1 作者侧

- 作者提供自己的母版能力
- 平台人工协助整理、审核和发布
- 母版只是作者赚钱的核心资产之一

### 7.2 买家侧

买家买到的是：

- 一个自动开通的实例
- 一个可直接使用的交付结果

买家默认不获得：

- 母版源内容
- 再发布权
- 再销售权
- 公开派生权

### 7.3 派生与侵权规则

当前原则：

- 默认禁止派生再卖
- 别人的模板不能直接 fork 后再售卖
- 买家购买的是使用权，不是发布权
- 未来隐私版模板中，买家看不到母版细节配置

## 8. Upgrade Model

`agent_manage` 未来需要支持母版升级，但不能简单覆盖用户实例。

### 8.1 升级应分层

建议至少分 3 层：

1. 母版基础层
2. 用户自定义层
3. 运行态数据层

运行态数据层不应参与母版升级。

### 8.2 必须记录的对象

后续应考虑至少记录：

- `template_release`
- `instance_base_version`
- `instance_overrides`
- `upgrade_job`

### 8.3 升级策略

当前倾向于保守升级：

- 只自动升级基础层
- 用户自定义层保持不动
- 无冲突部分自动升级
- 对 `soul / 配置` 冲突可考虑 AI 辅助整理
- 对 `skills / 程序` 冲突不应默认自动合并

### 8.4 AI 的合理角色

AI 更适合：

- 分析旧母版和新母版差异
- 分析用户改动
- 生成升级摘要
- 给出合并建议

AI 不适合默认自动处理复杂代码和脚本升级。

---

## 9. Key MVP Constraints

当前阶段明确不做：

- 普通作者自助上传整包即发售
- 普通用户把运行实例直接变成公开商品
- 买家购买后可再发布母版
- 自由继承链和分账链
- 完整开放式模板市场

当前阶段建议先做：

- 平台人工代发布作者母版
- 商品绑定审核后的 `template release`
- 支付成功后自动创建实例
- 买家进入 `My Claw`
- `agent_manage` 负责实例创建、渠道绑定、恢复和后续升级基础能力

## 10. Open Questions

以下内容后续需继续细化：

- `create_instance` 到底要复制或生成哪些文件
- skills 资产如何独立打包、版本化和审核
- 微信 / TG 绑定后到底需要 reload 还是 restart
- 模板升级时，用户改过的 soul / skills 如何存成 override
- 恢复备份是否长期都只恢复配置，是否未来扩展到 workspace 内容
- `My Claw` 的 list 页面结构
- 续费和过期后的恢复体验
